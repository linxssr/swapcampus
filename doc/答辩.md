## 一、软件开发生命周期（SDLC）——项目如何一步完成

### 第一阶段：可行性分析

**问题域分析：** 校内闲置交易依赖微信群，存在四个核心痛点：
- 信息散乱，无结构化搜索
- 匿名交易，信用无保障
- 沟通在第三方平台，无法追溯
- 无交易流程管理，纠纷无处理机制

**可行性结论：**

| 维度 | 结论 |
|------|---|
| 技术可行性 | Spring Boot + Vue3 + MySQL均为成熟栈，团队具备能力 |
| 经济可行性 | 使用免费开源组件，云服务器成本极低 |
| 操作可行性 | Web 端无需安装，校内同学即开即用 |
| 时间可行性 | 6 大模块按人员分工，8 周可交付 |

### 第三阶段：系统设计

#### 3.1 架构设计——前后端分离三层架构

```
┌─────────────────────────┐
│  前端表现层（Vue 3 + Element Plus）       │
│  用户端 SPA  ←→  管理员端 SPA            │
│  Axios HTTP / WebSocket 长连接           │
└────────────┬────────────────────────┘
               │ REST API  /api/v1/**
               │ WebSocket /ws/chat
┌────────────▼──────────────────────────┐
│  后端业务层（Spring Boot 3）              │
│  JWT 拦截器 → 6 大业务模块               │
│  M1用户 M2商品 M3检索                │
│  M4订单 M5聊天 M6管理                    │
│  ← MinIO 对象存储（图片）→               │
└──────────────┬────────────────────┘
               │ MyBatis
┌──────────────▼────────────────────────┐
│  数据持久层（MySQL 8.0）                  │
│  15 张数据表，含索引优化                  │
└────────────────────────────────┘
```

**架构决策理由：**
- 前后端分离：前端 Vite 独立部署，后端 JAR 独立部署，职责清晰，方便分工
- Maven 多模块：`swapcampus-common / model / mapper` 作为共享层，6个业务模块各自独立，低耦合
- MyBatis 而非 JPA：SQL 可见，复杂联查（商品列表 JOIN 分类 JOIN 用户）易控制

#### 3.2 数据库设计

**15 张核心数据表：**

| 表名 | 作用 | 关键设计 |
|------|------|
| `user` | 用户账号 | `student_id` UNIQUE，`credit_score` 默认 100 |
| `item` | 商品 | `audit_status`（0待审/1通过/2驳回）+ `publish_status` 双状态控制 |
| `item_image` | 商品多图 | `sort` 字段，首图即封面 |
| `orders` | 订单 | `order_no` UUID 唯一编号，`trade_type`（1面交/2柜取） |
| `chat_message` | 聊天 | `is_read` 未读标记，`msg_type`（文字/图片/表情） |
| `sign_record` | 签到 | `continuous_days` 连续天数，`UNIQUE(user_id, sign_date)` |
| `task_config` | 任务配置 | 静态配置表，`task_type`（1每日/2成就）|
| `user_task` | 任务进度 | `status`（0未完成/1待领取/2已领取） |

**索引设计（答辩时说）：**
```sql
-- 加速学号登录
CREATE UNIQUE INDEX idx_student_id ON user(student_id);
-- 加速商品列表查询（高频）
CREATE INDEX idx_item_categoryid ON item(category_id);
CREATE INDEX idx_item_audit_status ON item(audit_status);
-- 加速聊天记录拉取（双向会话）
CREATE INDEX idx_chat_from_to ON chat_message(from_uid, to_uid);
```

#### 3.3 接口设计（RESTful 规范）

统一前缀 `/api/v1`，统一响应格式：
```json
{ "code": 200, "message": "成功", "data": { ... } }
```

**鉴权机制：**
- 用户登录 → 返回 JWT Token（有效期 24h）
- 前端所有请求 Header 携带 `Authorization: Bearer <token>`
- 后端 `JwtInterceptor` 拦截，解析 userId 注入请求上下文
- 管理员独立 JWT，无法访问用户接口

#### 3.4 模块详细设计（以 M5 WebSocket 为例，体现难点攻克）

```
客户端连接：ws://localhost:8080/ws/chat?token=xxx
         ↓
ChatServer.onOpen()  解析 token → 获取 userId → 存入连接池 Map<Long, Session>
         ↓
ChatServer.onMessage()  接收 JSON 消息 → 持久化入库 → 查找接收方 Session → 推送
         ↓
ChatController.history()  REST 接口拉取历史消息（分页）
```

### 第四阶段：编码实现

**编码规范与工程实践：**

| 实践 | 具体做法 |
|------|------|
| 分层架构 | Controller → Service → Mapper，严格禁止跨层调用 |
| 统一异常处理 | `GlobalExceptionHandler` 捕获所有异常，返回标准 Result |
| DTO/VO 分离 | 请求用 DTO（防止过度绑定），响应用 VO（隐藏敏感字段） |
| 密码安全 | BCrypt 加盐哈希，不可逆，数据库不存明文 |
| 图片存储 | MinIO 对象存储，后端只存 URL，前端通过 Vite 代理访问（解决 CORS） |
| 前端状态管理 | Pinia 持久化 token，Axios 拦截器自动注入，图片 URL 自动代理转换 |

**关键技术难点（答辩时重点讲）：**

1. **WebSocket + JWT 鉴权**：WS 握手阶段无法设置 Header，改为 URL 参数传token，`onOpen` 时验证

2. **图片跨域访问**：MinIO 运行在 `localhost:9000`，浏览器 ORB 策略拦截。解决方案：Vite 配置 `/minio` 代理，`toProxiedUrl()` 工具函数统一转换所有图片 URL

3. **多模块 Maven 依赖**：`swapcampus-mapper` 同时被 6 个业务模块依赖，通过根 POM `dependencyManagement` 统一版本，避免冲突

4. **商品审核双状态**：`audit_status`（审核状态）和 `publish_status`（上架状态）分离，支持"审核通过但手动下架"场景

## 二、CI/CD 流程说明

### 本项目实际的 CI/CD 实践

虽然课设不强制要求全套 CI/CD 平台，但项目遵循了 CI/CD 的核心思想：

#### 持续集成（CI）——本地实践

```
代码提交
    ↓
mvn clean compile          # 编译检查，保证代码可构建
    ↓
mvn test                # 执行 JUnit 单元测试
    ↓
ReadLints（ESLint）         # 前端代码规范检查
    ↓
vue-tsc --noEmit           # TypeScript 类型检查
    ↓
npm run build              # 前端构建，验证无编译错误
```

每次修改代码后都执行上述步骤，确保主干始终处于可运行状态。

#### 持续交付（CD）——可描述的标准流程

如果接入 GitHub Actions，流水线设计如下：

```yaml
# .github/workflows/ci.yml（标准设计）

trigger: push to main / pull_request

jobs:
  backend-ci:
    steps:
      - mvn clean test          # 编译 + 单元测试
      - mvn package -DskipTests # 打包 JAR

    steps:
      - npm ci                  # 安装依赖（精确版本）
```

## 三、软件工程核心知识点对照

### 设计原则落地情况

| 原则 | 在本项目中的体现 |
|------|-------------|
| 单一职责 SRP | Controller 只做参数校验和响应封装，业务逻辑全在 Service，SQL 全在 Mapper |
| 开闭原则 OCP | `task_config` 任务配置表静态化，新增任务只需插入一条记录，无需改代码 |
| 依赖倒置 DIP | Controller 依赖 Service 接口（`UserService`），不直接依赖 `UserServiceImpl` |
| 接口隔离 ISP | `UserService` 只声明用户相关方法，`SearchService` 只声明检索相关方法，不混用 |
| DRY（不重复） | 公共 JWT 校验、MinIO 上传、Result 包装抽取到 `swapcampus-common`，6个模块复用 |

### 设计模式应用

| 模式 | 位置 | 说明 |
|------|------|
| 模板方法 | `JwtInterceptor.preHandle()` | 统一处理所有需要鉴权的接口 |
| 工厂模式 | Spring IoC 容器 | Bean 的创建和注入由容器管理 |
| 单例模式 | `@Service / @Component` Bean | Spring 默认单例作用域 |
| 代理模式 | `@Transactional` | Spring AOP 动态代理实现事务 |
| VO/DTO 模式 | `ItemDetailVO`, `ItemPublishDTO` | 隔离数据库实体与接口数据，防止字段泄露 |

### 非功能需求实现情况

| 非功能需求 | 目标 | 实现方式 | 达成情况 |
|-----------|------|
| 性能 | WebSocket 延迟 ≤ 200ms | 点对点推送，消息不经过轮询 | ✅ 局域网 < 10ms |
| 安全 | 密码加密 | BCrypt 加盐哈希 | ✅ |
| 安全 | 接口鉴权 | JWT + 拦截器 | ✅ |
| 安全 | 学号实名 | `student_real` 对照表校验 | ✅ |
| 数据量 | ≥ 200 条商品 | `seed.sql` 注入 200+ 条 | ✅ |
| 可维护性 | 模块化 | Maven 多模块，业务独立 | ✅ |
| 可扩展性 | 加分项功能 | 签到/任务/推荐独立模块 | ✅ 均已实现 |

## 四、答辩高频问题参考答案

**Q：为什么选择前后端分离架构？**

A：分离后前端 Vue 和后端 Spring Boot 可以独立开发、独立测试、独立部署。团队成员可以并行工作，前端不需要等后端完成才能联调，约定好接口文档后用 mock 数据先开发。同时前端产物是纯静态文件，可部署到 CDN，性能更好。

**Q：JWT 和 Session 鉴权有什么区别，为什么选 JWT？**

A：Session 是有状态的，服务器需要存储会话信息，横向扩展时需要共享 Session（Redis）。JWT 是无状态的，服务器不存任何信息，token 本身携带用户信息，验签即可。本项目是单服务器部署，两者都可以，选 JWT 是因为它更标准，与 WebSocket 鉴权也能统一（WS 握手时 URL 传token）。

**Q：WebSocket 如何处理用户下线？**

A：`ChatServer.onClose()` 回调触发时，从连接池 `Map<Long, Session>` 中移除对应的 userId。推送消息时先判断 Session 是否存在，不存在说明对方离线，消息仍然入库，用户下次上线通过 REST 接口拉取历史消息。

**Q：商品图片为什么不直接存路径，而要用 MinIO？**

A：数据库只存 URL 字符串，实际图片文件放在 MinIO 对象存储。这样做的好处：① 数据库表体积小，不存二进制 BLOB；② MinIO 支持直接 HTTP 访问，前端可以直接请求图片 URL；③ 对象存储天然支持大文件、高并发读取，比数据库性能高很多。

**Q：信用积分如何保证不被刷？**

A：三道防线。① 签到用 `UNIQUE KEY(user_id, sign_date)` 数据库约束，同一天重复插入直接报错；② Service 层先查再插，已签到则抛 `RuntimeException`；③ 任务领取用 `UPDATE ... WHERE status = 1`，CAS 更新，只有状态为"待领取"时才能领，领过的更新不到。

**Q：为什么要分 audit_status 和 publish_status 两个字段？**

A：`audit_status` 表示审核流程状态（待审/通过/驳回），`publish_status` 表示上架状态（上架/下架）。两者独立是因为存在这样的业务场景：商品审核通过后（`audit_status=1`），如果被举报违规，管理员可以直接将 `publish_status=0 下架，而不需要改变审核记录。这样的设计也方便将来增加"卖家手动下架再上架"功能，无需重新走审核流程。
